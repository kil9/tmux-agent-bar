# AGENTS.md

문제 정의 → 작고 안전한 변경 → 변경 리뷰 → 리팩터링 — 이 루프를 반복한다.

## 마인드셋

- 시니어 엔지니어처럼 생각한다.
- 추측으로 뛰어들거나 성급히 결론내리지 않는다.
- 항상 여러 접근을 평가하고, 장점/단점/위험을 각각 한 줄로 적은 뒤 가장 단순한 해법을 선택한다.
- 선택지가 불분명한 경우 사용자에게 선택 가능한 대안을 설명과 함께 제시하고 선택하게 한다.

## 필수 코딩 규칙

- 코딩 전에 Problem 1-Pager: 배경 / 문제 / 목표 / 비목표 / 제약을 작성한다.
- 명시적인 코드를 선호한다; 숨겨진 "매직" 금지.
- DRY를 따르되, 섣부른 추상화는 피한다.
- 구체적으로 예외를 처리하고, 사용자에게 명확한 메시지를 제공한다.
- 관측 가능한 API 동작 변경이 있을 경우 항상 사용자에게 확인할 수 있는 방식을 제공한다.

## 코드 및 파일 참조 규칙

- 코드를 변경하기 전에 정의, 참조, 호출 지점, 관련 테스트, 문서/설정/플래그를 찾아 읽는다.
- 심볼을 수정하기 전에 전역 검색으로 사전/사후 조건을 파악한다.

## 코딩 스타일 및 네이밍 규칙

- 프로젝트의 주 언어/프레임워크 관용 스타일을 따른다.
- 포매터와 import 정리 도구를 우선 적용한다(예: `go fmt`, `goimports`, `prettier`, `ruff format`).
- 줄 길이는 합리적으로 유지하고, 함수는 짧고 응집력 있게 유지한다.
- 네이밍은 의도를 드러내고, 약어는 팀 컨벤션에서 허용한 경우에만 사용한다.

## 테스트 규칙

- 새 코드에는 새 테스트를 추가한다; 버그 수정에는 회귀 테스트를 반드시 포함한다(먼저 실패하도록 작성).
- 테스트는 결정적이고 독립적이어야 한다.
- E2E 테스트에는 1개 이상의 성공 경로와 1개 이상의 실패 경로를 포함한다.
- 동시성/락/재시도에서 비롯될 위험(중복, 데드락 등)을 선제적으로 평가한다.
- 경계값/실패 경로를 기본으로 포함한다.
- 가능하면 테이블 기반 테스트를 우선 검토한다.

## 보안 규칙

- 코드/로그/티켓에 비밀값을 절대 남기지 않는다.
- 입력을 검증·정규화·인코딩하고, 파라미터화된 접근을 사용한다.
- 최소 권한 원칙을 적용한다.
- 시크릿/토큰/인증정보는 커밋하지 않는다.

## 클린 코드 규칙

- 의도를 드러내는 이름을 사용한다.
- 각 함수는 한 가지 일만 한다.
- 상수는 항상 심볼화한다(하드코딩 금지).
- 코드를 입력 → 처리 → 반환 구조로 구성한다.
- 실패는 구체적인 오류/메시지로 보고한다.
- 테스트는 사용 예제로도 동작하게 하고, 경계/실패 사례를 포함한다.

## 안티 패턴 규칙

- 문맥을 읽지 않고 코드를 수정하지 않는다.
- 실패나 경고를 무시하지 않는다.
- 단순히 컨벤셔널하다는 이유로 근거 없는 최적화나 추상화를 도입하지 않는다.
- 광범위한 예외를 남용하지 않는다.

## 커밋 및 풀 리퀘스트 규칙

- 사용자의 명시적인 요청이 있기 전까지는 커밋하지 않는다.
- 커밋을 요청받았을 때는 `AGENTS.md`, `README.md`, `PLAN.md`의 업데이트가 필요한지 검토하고, 필요하면 함께 업데이트한다.
- 기본 브랜치에 직접 푸시하지 않고 PR을 기본 절차로 사용한다.
- 새로운 PR은 항상 Draft로 생성한다.
- PR 전 린트/테스트를 통과시키고, 변경 이유와 범위를 명확히 작성한다.
- 팀 규칙이 있다면 이슈 번호 연결, 리뷰어, 체크리스트를 포함한다.
- 커밋 메시지는 한국어로 작성한다.
- 커밋 메시지 첫 줄에는 80자 이내로 전체 커밋 내용을 요약한다.
- 두 번째 줄은 비워둔다.
- 세 번째 줄부터 목록 형태로 간단하게 변경 내용을 기록한다.
- 커밋 메시지를 길게 작성해야 할 필요가 있는 경우 목록 아래에 다시 한 줄을 띄우고 기록한다.

## 이 저장소 전용 정보

### 프로젝트 개요

`tmux-agent-bar`는 tmux의 기존 윈도우 표시 영역에 이모지를 삽입하여 Claude Code 에이전트 상태를 시각화하는 도구다. 별도의 상태 바(status bar)를 추가하지 않는다.

### 핵심 동작 원리

1. tmux 세션의 모든 윈도우·pane을 순회한다.
2. 각 pane에서 Claude Code 프로세스 실행 여부 및 상태(승인 대기 / 완료 / 실행 중)를 감지한다.
3. `tmux rename-window` 또는 tmux format hook을 이용해 윈도우 이름 앞에 이모지를 삽입한다.

**상태 우선순위**: `⏳` (승인 대기) > `✅` (전체 완료) > 이모지 없음 (실행 중 또는 없음)

### Claude Code 상태 감지 방법

- **Claude Code hooks**: `~/.claude/settings.json`의 `Stop` / `Notification` hooks에서 `tmux-agent-bar hook <status>` 호출
- 상태는 `/tmp/tmux-agent-bar/<session>_<window>_<pane>` 파일에 기록됨

### 프로젝트 구조

```
tmux-agent-bar/
├── README.md
├── AGENTS.md
├── PLAN.md
├── go.mod
├── main.go        # 엔트리포인트 + 모든 로직
└── main_test.go   # 집계 로직 단위 테스트
```

### 빌드/실행/테스트 명령어

```bash
go build -o tmux-agent-bar .        # 빌드
go test ./...                       # 테스트
./tmux-agent-bar hook waiting       # 현재 pane 상태를 waiting으로 기록
./tmux-agent-bar hook done          # 현재 pane 상태를 done으로 기록
./tmux-agent-bar status <window>    # 윈도우 이모지 반환
```

### 환경변수 및 런타임 플래그

- `TMUX_PANE`: tmux가 설정하는 현재 pane ID (예: `%3`). hook 실행 시 필수.
- `TMUX`: tmux 소켓 경로. 존재하면 tmux 세션 내로 판단.
- 상태 파일 디렉토리: `/tmp/tmux-agent-bar/` (하드코딩, 변경 불가)

### 보안 체크리스트

- tmux 소켓에 대한 접근은 현재 사용자 권한 범위 내로 제한한다.
- pane 내용 캡처 시 민감 정보(API 키 등)가 로그에 남지 않도록 주의한다.
- 외부 프로세스 실행 시 셸 인젝션을 방지한다.
